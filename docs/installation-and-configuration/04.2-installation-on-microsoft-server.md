# Imperva Agent 14.9 Deployment — MS SQL Server

### Agent Installation & IPC Interface Configuration

**Target:** Windows Server 2012 R2 (MS SQL Server 2014)
**Imperva Gateway Version:** 14.19.1.80_0
**Agent Version:** 14.9

---

## Overview

This procedure documents the deployment of Imperva Agent v14.9 on an MS SQL Server 2014 host, covering the database-side prerequisite login, agent installation, logical mapping in the MX Console, and binding of the `MssQL IPC` interface to capture local SSMS traffic that runs over Shared Memory rather than TCP.

---

## 0. Database Prerequisites — Agent User Creation

**Run on the target MS SQL instance, in the `master` database:**

```sql
-- Switch to the master database to create the server-level login
USE master;
GO

-- Create the login for the Imperva agent
CREATE LOGIN imperva_monitor WITH PASSWORD = 'YourSecurePasswordHere';
GO

-- Grant the required server-level permissions for passive monitoring
GRANT VIEW ANY DEFINITION TO imperva_monitor;
GRANT VIEW SERVER STATE TO imperva_monitor;
GO
```

> **Note:** These are server-level, read-only visibility permissions. They allow the agent to resolve object and session metadata for audit purposes — they do **not** grant access to user data.

---

## 1. Installation

**Navigation:** Run installer as **Administrator** on the Windows VM

1. Install Imperva Agent **v14.9**
2. Configure the installation for **Local** traffic monitoring
3. Point the agent to Gateway IP: `192.168.10.243`

---

## 2. Logical Mapping in MX Console

**Navigation:** `Main > Setup > Sites / Agents`

1. Add the Windows VM (`192.168.10.247`) as a **Server** under the `windows_server_db` Server Group
2. Assign the `mssql_agent_01` agent to the `windows_server_db` Server Group via the agent's **Settings** tab
3. Enable **Agent Monitoring** on the `ms_sql_2014` service

---

## 3. Capturing Local SSMS Traffic — IPC Interface

Local queries run through SQL Server Management Studio (SSMS) on the database server use **Shared Memory**, not TCP, so they bypass the agent's standard network sniffer. Capturing this traffic requires a dedicated IPC interface.

**Navigation:** `Main > Setup > Agents > mssql_agent_01 > Data Interfaces tab`

1. Add a new **Data Interface** of type `MssQL IPC`
2. Bind the IPC interface to the `ms_sql_2014` service
3. Set the **Server Path** to the named pipe:

```
\\.\pipe\sql\query
```

> **Why this matters:** This is the MS SQL equivalent of Oracle BEQ/IPC binding. Network-based sniffers only see remote connections — a DBA or application using SSMS directly on the database server communicates over Shared Memory, which stays invisible to the agent until the `MssQL IPC` interface is explicitly bound to the service.

---

## 4. Verification

**Navigation:** `Main > Audit > DB Audit Data > Data view`

1. Run test `SELECT` queries in SSMS on the local server
2. Set **Time Frame** to `Last Hour`
3. Confirm captured queries and login events appear in the audit data view ✔
