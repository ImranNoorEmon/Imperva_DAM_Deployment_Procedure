## Agent Installation on Linux — DPE Deployment

The deployment was performed on a **designated test database environment** rather than the live IPEMIS production servers. This approach allows policy tuning, baseline establishment, and verification before any production change.

### Phase 0 — Agent Installation on the Database Server

**Step 1: Download the reconnaissance script**
```bash
# Access Imperva FTP server
# Navigate to: /misc/
# Download: which_ragent.sh
```

**Step 2: Identify the correct agent package**
```bash
chmod +x which_ragent.sh
./which_ragent.sh
# Script analyses OS version, kernel version, and architecture
# Outputs the exact agent package name compatible with this system
```

**Step 3: Retrieve the agent package**
```
# Return to Imperva FTP
# Download: the recommended .bsx binary package
# Download: the corresponding KABI file (if required by kernel version)
```

**Step 4: Set installer file ownership**
```bash
chown root:root install.sh
```

**Step 5: Configure Secure Boot support (if applicable)**

> **Prerequisite:** `mokutil` must be installed. It is available through the `mokutil` package — refer to your database vendor's documentation for installation guidance.

If the target server has UEFI Secure Boot enabled, the Imperva kernel module key must be enrolled **before** the agent is installed, otherwise the kernel will block the driver.

1. Unzip the SecureSphere Agent installation package and run the following in the extraction directory:
   ```bash
   mokutil --import public_key_OEL-v7-kUEK-v3-px86_64.der
   ```
   > **Note:** The filename may differ depending on the agent version. This extracts the public key and prompts for a password.

2. Set a password and note it down — it will be required in the next step. Open a console to the database server and reboot it. Do not press any key until the server comes up and the automated UEFI key enrollment screen appears — you have only **10 seconds** to press any key to enter the UEFI key enrollment process. Follow the wizard's steps.

3. When prompted, enter the password created in step 2. Once complete, the key is registered and the server will reboot. Wait for the server to come back up — do **not** re-enter the BIOS or UEFI screen.

4. Verify the Imperva key appears in the keyring:
   ```bash
   mokutil --list-enrolled | grep Imperva
   ```
   Expected output:
   ```
   [root@localhost tmp]# mokutil --list-enrolled | grep Imperva
        Issuer: O=Imperva Inc., CN=Imperva Inc./emailAddress=support@imperva.com
        Subject: O=Imperva Inc., CN=Imperva Inc./emailAddress=support@imperva.com
   [root@localhost tmp]#
   ```

5. Once the output is verified:
   - If the agent is **not** installed, proceed to install it now (Step 6 below)
   - If a SecureSphere Agent that supports Secure Boot (**v13.2 or newer**) is already installed, simply restart the agent

**Step 6: Execute the installer**
```bash
# Run as root
./install.sh
# OR
./agent_package.bsx
# Provide the KABI file path when prompted (Linux/Oracle deployments)
# The installer hooks the agent into the OS kernel
```

**Step 7: Configure the agent to point to the Gateway**
```bash
# During or after installation, configure:
# Gateway IP address: <gateway_ip>
# Gateway Port: 443
# SSL: enabled
# This establishes the secure forwarding tunnel
```

---

### Agent Registration & Connectivity — Summary

- **Ports:** The agent only requires an **outbound** connection to the Gateway on **TCP 443 (SSL)**. No inbound ports need to be opened on the database server for the agent itself.
- **Configuration files:** The agent's connection to the Gateway is defined in `cov_proxy.xml` (Gateway host, port, and SSL flag, plus the agent's registered name). If a registration attempt doesn't complete automatically, a `registration.xml` file can be used to mark the agent as registered so the service starts cleanly.
- **What the agent needs to do:** Register itself with the Gateway (either interactively, via the `ragentctrl register` CLI command, or manually through the config files above), then run persistently as a `systemd` service so it survives reboots and restarts on failure.
- **Before reinstalling or re-registering:** Stop and clear any stale agent processes and remove leftover lock files to avoid registration conflicts.

### Enable Persistence & Start the Agent Service

```bash
systemctl daemon-reload
systemctl enable ragent
systemctl start ragent
```

Final status check:
```bash
systemctl status ragent
```

### Database User Configuration (Oracle)

Create the monitoring user inside the Oracle Database so the agent can see activity.

```bash
su - oracle
sqlplus / as sysdba
```

Ensure the database and PDB are open:
```sql
STARTUP;
ALTER PLUGGABLE DATABASE ORCLPDB OPEN;
ALTER PLUGGABLE DATABASE ORCLPDB SAVE STATE;
-- (If STARTUP says "already open," ignore it and run the next lines)
```

Switch to the PDB:
```sql
ALTER SESSION SET CONTAINER = ORCLPDB;
```

Run the user creation script:
```sql
CREATE USER imperva_monitor IDENTIFIED BY "StrongPassword123!";
GRANT CREATE SESSION TO imperva_monitor;
GRANT SELECT ON V_$SESSION TO imperva_monitor;
GRANT SELECT ON V_$SQL TO imperva_monitor;
GRANT SELECT ON V_$SQLTEXT TO imperva_monitor;
GRANT SELECT ON V_$SQLAREA TO imperva_monitor;
GRANT SELECT ON DBA_USERS TO imperva_monitor;
GRANT SELECT ON DBA_AUDIT_TRAIL TO imperva_monitor;
```
